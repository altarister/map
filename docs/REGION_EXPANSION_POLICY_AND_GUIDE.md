### 0.2 핵심 데이터 구조 확인 (광역 자치단체/시·군·자치구/읍·면·법정동 계층 시스템)

우리의 지도 정책은 대한민국의 실제 통계청 행정구역 분류 체계(자릿수)를 그대로 따릅니다.

1. **광역 자치단체**: 앞 2자리 (`11` 서울, `23` 인천, `31` 경기 등)
2. **시/군/자치구**: 앞 5자리 (`11010` 종로구, `41110` 수원시 등)
3. **읍/면/법정동**: 7자리~8자리 (`1101053` 사직동 등)

`merged_map.geojson` (VWorld 원본)의 properties 필드:

```json
{
  "code": "41610101",       ← 8자리: 읍/면/동 그룹 (퀴즈 출제 대상)
  "name": "경안동",
  "SIG_KOR_NM": "광주시",   ← 시군구 이름 (구 있는 시 판별 핵심 필드!)
  "EMD_KOR_NM": "경안동",
  "_isEmdGroup": true
}

{
  "code": "4161025025",    ← 10자리: 리(里) 단위 (출제 제외)
  "name": "산이리",
  "SIG_KOR_NM": "광주시"
}
```

### 0.3 행정구역 유형 자동 감지 — 구현 가능성: ✅ 완전 가능

**`SIG_KOR_NM` 필드의 공백(space) 패턴으로 100% 구별 가능:**

| 패턴 | 예시 | 해석 |
|------|------|------|
| `"SIG_KOR_NM": "광주시"` | 공백 없음 | 구 없는 시 |
| `"SIG_KOR_NM": "가평군"` | 공백 없음, `군`으로 끝남 | 군(郡) |
| `"SIG_KOR_NM": "용인시 처인구"` | 공백 있음, 뒤에 `구` | 구 있는 시 |

```typescript
// 완전히 구현 가능한 코드 (데이터에서 검증됨)
function detectAdminType(sigKorNm: string): AdminUnitType {
  if (sigKorNm.includes(' ')) return 'GU_CITY';  // "용인시 처인구"
  if (sigKorNm.endsWith('군')) return 'GUN';       // "가평군"
  return 'PLAIN_CITY';                              // "광주시"
}
```

## 1. 문제 정의 (Problem Statement)

### 1.1 핵심 충돌

기존 GDD의 코스 설계(기본 코스 / 상세 코스)는 **행정구역 진입 단계가 일정하다는 암묵적 가정** 위에 세워져 있습니다. 그러나 대한민국 행정구역은 도시마다 계층 단계가 다릅니다.

## 2. 대한민국 행정구역 통합 분류 체계 (Systematic Hierarchy)

대한민국의 행정구역은 다음과 같은 **3단계 구조**의 데이터 코드로 완벽하게 계층화되어 있으며, 게임도 이 위상을 따릅니다.

* **[광역 자치단체] (코드 2자리)**
  * 예: 서울(`11`), 인천(`23`), 경기(`31`)
  * **UI 취급**: 첫 진입 시 화면에 깔리는 가장 큰 땅입니다. (`vworld_provinces.geojson` 사용)

* **[시/군/자치구] (코드 5자리)**
  * 예: 강남구(`11230`), 부평구(`23060`), 용인시 기흥구(`31192`)
  * **핵심 예외 처리 (수원/용인/고양 등 대도시 "일반구" 문제)**
    * 데이터 원본에는 "용인시 전체"(`41460`) 폴리곤이 시/군/자치구 일반 구 단위로 찢어져 있습니다.
    * 하지만 전국 스케일(경상북도, 전라남도 등) 확장을 고려했을 때, 포항시 남/북구나 전주시 완산/덕진구 등을 처음부터 평면(Flat)에 노출하면 **유저의 지리적 상식(도 -> 시 -> 구)과 크게 충돌**합니다.
    * 따라서 시스템 및 UI 관점에서는 쪼개진 원본 데이터(처인구, 기흥구)를 코드로 강제 병합하여 **"용인시" 전체를 대표하는 단일 [시/군/자치구] 단위로 묶어서 취급**합니다.
  * **UI 취급**: [광역 자치단체]의 땅을 누르면 나타나는 하위 구역. "기본 코스"의 무대.

* **[대도시 일반구] (코드 5자리, 선택적)**
  * 예: 덕양구(`41281`), 기흥구(`41464`), 처인구(`41463`)
  * **조건**: `SIG_KOR_NM`에 공백이 있는 대도시(고양시, 수원시, 용인시 등)만 해당
  * **UI 취급**: CITY 단계에서 대도시 클릭 시 삽입되는 중간 선택 화면. 없으면 바로 DONG으로 진입.

* **[읍/면/법정동] (코드 7~8자리)**
  * 예: 역삼동(`1123064`), 산이리 등은 읍/면 그룹(`_isEmdGroup`)으로 병합.
  * **UI 취급**: 최종 퀴즈의 타겟들. (가장 말단이므로 리 단위는 영구 배제)

### 2.1 SelectionLevel — 코드 타입 정의 (`src/types/region.ts`)

위 행정 계층은 코드에서 `SelectionLevel` 타입으로 1:1 대응됩니다:

```ts
export type SelectionLevel =
  | 'PROVINCE'   // 광역 자치단체 (서울/인천/경기, 코드 2자리)
  | 'CITY'       // 시/군/자치구 (수원시/광주시, 코드 5자리)
  | 'DISTRICT'   // 대도시 일반구 (덕양구/기흥구, 코드 5자리, 조건부)
  | 'DONG';      // 읍/면/법정동 (게임 진입, 코드 7~8자리)
```

**진입 경로:**

| 도시 유형 | 선택 흐름 |
|---|---|
| 일반 시/군 (광주시, 가평군) | `PROVINCE` → `CITY` → *게임 시작* |
| 대도시 특례 (고양시, 수원시) | `PROVINCE` → `CITY` → `DISTRICT` → *게임 시작* |
| 특별·광역시 자치구 (서울 강남구) | `PROVINCE` → `CITY` → *게임 시작* |

`GameContext.selectionLevel`로 현재 단계를 추적하며, 이전의 `selectionDepth: number`(1~3)를 대체합니다.

### 도시 유형별 화면 진입 해석 (전국구 유저 상식 우선 정책 - A안)

**전국구 확장을 위한 표준화 타겟**
대도시의 하위 일반구를 처음부터 화면에 쫙 뿌리면(Flat), 유저는 광역 자치단체 바로 밑에 생소한 구 이름들을 맞닥뜨려 혼란을 겪게 됩니다. 따라서 **대한민국의 정식 행정구역 체계(유저 상식)를 UI 화면 진입 순서로 그대로 적용**합니다.

**1) 서울 / 인천 / 부산 등 (특별·광역시 형)**
- [광역 자치단체] 클릭: "서울특별시" (11) 선택
- [시/군/자치구] 화면: 강남구, 서초구, 마포구 등 **분리된 자치구들이 한 화면에 렌더링**됨 (정상)
- [단일 구 클릭 시] 퀴즈: 강남구 진입 시 곧바로 "역삼동", "삼성동" 출제

**2) 용인 / 수원 / 포항 / 전주 등 (일반구가 있는 특례시·대도시 형 - 일반구 선택 화면 추가)**
- [광역 자치단체] 클릭: "경기도" (31) 
- [시/군/자치구] 화면: 기흥구, 처인구가 직접 노출되지 않고 묶여서 **"용인시" 단일 버튼/폴리곤으로 렌더링**됨. (포항시 역시 남/북구가 하나로 묶임)
- [단일 시 클릭 시] 상세 구역 선택창: "용인시 기흥구", "용인시 처인구" 중 하나를 고르는 **중간 구 선택 화면 삽입**
- [구 클릭 시] 퀴즈: 기흥구 진입 시 "보라동", "상갈동" 출제

**3) 광주 / 가평 / 구미 등 (구 없는 일반 시·군 형)**
- [광역 자치단체] 클릭: "경기도" (31) 
- [시/군/자치구] 화면: 광주시, 가평군 단일 노출
- [단일 시/군 클릭 시] 퀴즈: 하위 구가 없으므로 중간 자치구 선택 화면을 생략하고 곧바로 "경안동", "오포읍" 출제

## 3. 리(里) 효용성 최종 판단

### 3.1 도로명주소에서의 리(里) 위상

```
[도로명 주소 표준]
경기도 광주시 초월읍 경충대로 11번길 34   ← '산이리'는 주소에 없음
```

- 도로명주소(2014년 전면 시행): **리를 명시하지 않음**
- 화물 배차앱: 도로명주소 기반 → **리 단위 노출 안 됨**
- 화물 기사 판단에 읍/면 수준이면 충분

### 3.2 데이터 관점에서도 리는 불필요

- 경기도 데이터에서 10자리 코드(리 단위) 1,398개 존재
- 하지만 **8자리(읍/면/동 그룹)로 이미 병합 처리가 완료**되어  
  `_isEmdGroup: true` 플래그가 붙어 있음
- 현재 게임 로직도 이미 8자리 코드만 사용 중

### 3.3 결론: 리(里) 전면 제외 확정

> ✅ **데이터에도 이미 이 방향으로 구현되어 있음. 정책과 데이터가 일치.**

최종 퀴즈 출제 단위 = [읍/면/법정동]  (배차 플랫폼 노출 기준)
```

### 4.2 코스 정의 (전국 표준화 룰)

| 코스명 | 타겟 스케일 | 설명 |
|--------|-----------|-----------|
| **[A] 광역 코스** | 시/군/자치구 | 광역 자치단체 단계에서 소속 시/군/자치구 조각들을 맞추는 훈련 |
| **[B] 지역 코스** | 읍/면/법정동 | 시/군/자치구 하나를 찍고, 그 안의 모든 실전 배차지(읍/면/법정동)를 맞추는 심화 훈련. ★ 핵심 |

### 4.3 코스 선택 UI 흐름 및 성능 최적화 가이드

**개발 부하(DOM 폭발 및 Topo-gap)를 최소화하기 위한 렌더링 원칙**을 기획 단에서 강제합니다.

```
[1. 광역 자치단체 선택 화면] 광역 자치단체 (서울, 인천, 경기 등) 렌더링 (폴리곤 최소화, 부하 0)
         ↓
         원하는 광역 버튼(Ex. 경기도) 클릭
         ↓
[2. 시/군/자치구 선택 화면] 시/군/자치구 (렌더링 부하 억제)
         - 선택한 광역 단위 하위의 [시/군/자치구] 폴리곤만 화면에 렌더링함.
         - ⚠️ 개발 수칙 1: 일반구를 소유한 거대 도시(용인, 수원 등)는 구 단위 조각 폴리곤을 런타임에 직접 합치려 하지 말고, **빌드 파이프라인이 미리 생성해 둔 "용인시" 1개 덩어리 원본(\`vworld_sig_merged.geojson\`)을 그대로 렌더링**함.
         ↓
         원하는 지역(Ex. 용인시) 클릭
         ↓
[3. 대도시 일반구 선택 화면] 일반구 선택 (⚠️ 대도시 전용)

         - (가평군처럼 하위 일반구가 없는 곳은 본 선택 화면을 생략하고 곧바로 퀴즈로 직행함)

## 9. 실무: 지역 추가 확장 매뉴얼 (VWORLD SSOT 파이프라인)

과거 여러 종류의 스크립트(`bake_nationwide...`, 파이썬 스크립트 등)를 혼용하던 아키텍처는 전면 폐기되었으며, 현재는 **단일 원본 (Level 3 데이터)에서 하위 레벨 데이터를 모두 추출하고, 위상(Topology) 정리를 수행하는 SSOT(Single Source of Truth) 스크립트 파이프라인**을 사용합니다.

### 9.1 데이터 파이프라인 (SSOT Architecture)

![Data Pipeline]
(VWORLD Data) → `merged_map.geojson` (원본)
  │
(Run Script)  → `node scripts/build_vworld_levels.cjs`
  │
 ├─ [Turf.js 병합] Level 2 일반 시군구 파생 (`vworld_sig.geojson`)
 ├─ [Turf.js 병합] Level 2 특례시 묶음 파생 (`vworld_sig_merged.geojson`)
 ├─ [Turf.js 병합] Level 1 광역시도 파생 (`vworld_provinces.geojson`)
  │
(Optimizations)
 ├─ Tiny Artifact(미세 구멍 < 50sq.km) Turf 필터링 제거 (지도 내부 잡티 먼지 제거)
 ├─ `Mapshaper` (-clean, -simplify 20~30%): 용량을 무려 80% 줄이고 해안선 아티팩트(Sliver)를 부드럽게 용접
 └─ `Turf.rewind(reverse: true)`: D3.js 렌더링 엔진 폭발(경기를 누르면 지구 전체가 클릭되는 무한 볼륨 버그) 방지를 위해 폴리곤 좌표를 GeoJSON의 Right-Hand Rule(반시계 방향) 기준에 강제 체인(Chain).

### 9.2 새 지역 추가 절차 (Tutorial)

**Step 1: 원본 구하기 (Level 3 추가)**
1. VWORLD에서 타겟 지역의 읍/면/동/리 폴리곤(Shapefile 등)을 확보합니다.
2. QGIS 등의 도구를 사용하거나 VWORLD API를 활용해 GeoJSON 형식으로 변환합니다.
3. 이를 `public/mapData/merged_map.geojson` 파일의 `features` 배열 안에 밀어 넣습니다. (이때 `code`, `name`, `SIG_KOR_NM` 등 기존 프로퍼티 포맷과 맞추어야 합니다).

**Step 2: 인텔리전스 데이터 작성 (옵션)**
- `scripts/data/intel/XXXXX_regionname.json` 에 문제 출제 힌트용 json 데이터를 작성합니다. (기존에 작성되어 있다면 통과)
- `node scripts/merge_intel_to_geo.js` 를 돌려 `merged_map.geojson` 내부에 `properties.intel` 이 합쳐지도록 만듭니다.

**Step 3: 단일 맵 빌더 스크립트 실행 (가장 중요)**
```bash
node scripts/build_vworld_levels.cjs
```
- 스크립트가 실행되면 터미널에 Level 1 단위부터 Level 2 자치구 단위들까지 모두 `turf.union` 을 수행하는 진행상황이 나옵니다.
- 마지막으로 `Mapshaper` 패키지를 호출하여 용량을 줄이고 찌꺼기 구멍(Artifact Holes)을 지우는 작업을 수행합니다.

**Step 4: 로컬 서버 확인 및 브라우저 강제 새로고침**
`npm run dev` 를 통해 게임화면을 켭니다. 
브라우저 캐시로 인해 오래된 Map이 보일 가능성이 높으므로 **Cmd + Shift + R** (완전 새로고침) 을 반드시 수행한 후 추가한 지역이 지도상에 뜨고, 클릭이 정상적으로 파티셔닝 되는지 체크합니다.

## 10. Stage 2 (실전 배차) 연동 시 지역 확장 고려사항

Stage 2(실전 배차 필터링)는 유저의 현위치(Current Location)를 기준으로 반경(예: 15km) 내의 상차지(출발지)를 탐색하고 배차 보드(UI)에 리스트를 렌더링해야 합니다. 
지역(데이터) 확장 시 다음 사항들이 반드시 보장되어야 게임 엔진(`generator.ts`)이 정상 작동합니다.

1. **중심점(Centroid) 필수:** 확장되는 모든 읍면동/시군구 리스트(GeoJSON)는 반드시 `properties.centroid` `[longitude, latitude]` 필드를 포함해야 합니다. (거리 필터링 계산 및 지도 애니메이션 렌더링에 핵심)
2. **시군구(SIG) 및 읍면동(EMD) 포맷팅 유지:** 배차 보드의 <출발지> 및 <도착지> 틱스트는 `SIG_KOR_NM` 및 `EMD_KOR_NM` (또는 `name`) 필드를 쪼개고 가공하여 "고양시 / 주교동" 같은 UI로 나타납니다. 신규 데이터 추가 시 해당 프로퍼티 네이밍 규칙이 누락되지 않도록 유의해야 합니다.
