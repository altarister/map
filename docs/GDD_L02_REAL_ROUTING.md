# 인성 배차 시뮬레이터: 현실적 라우팅 고도화(Real Routing) 기획안

## 1. 개요
현재의 `RouteOptimizer`는 콜 간의 순서 최적화(TSP)를 계산할 때, 단순히 지표면의 위/경도를 이은 **하버사인(Haversine) 최단 직선거리**를 기준으로 운임과 우회율을 산출하고, 맵에는 5각형 꺾은선을 애니메이션으로 그리고 있습니다.
하지만 현실의 배차 및 합짐은 *산을 넘지 못하며, 고속도로 유무에 따라 전략적 경로가 확연히 달라집니다.*
본 문서는 이 한계를 극복하고 시뮬레이터의 현실성을 극대화하기 위해 논의된 두 가지 고도화 방법론을 정의합니다.

---

## 2. 접근 방법론 비교

### 방법 A: 인텔 메타데이터 기반 스마트 휴리스틱 (Intel Heuristic Routing)
- **개념:** 거대한 네트워크 망 연산 없이, 기존에 구축된 지역별 `RegionIntel` 데이터(배달의 핵심 정보) 내 `roads`(주요 간선/고속도로 목록) 항목을 활용하는 방식.
- **핵심 로직:**
  - 합짐의 가장 긴 동선(Trunk line)인 **마지막 상차지 -> 첫 번째 하차지**를 추출.
  - 양쪽 지역의 `intel.roads` 배열을 탐색하여 **교집합(공유하는 고속도로/간선도로)** 이 존재하는지 판별.
  - 교집합 도로(예: "수도권제1순환고속도로")가 존재하면 트래픽과 도심을 우회할 수 있는 **[최고의 합짐 동선]** 으로 간주하여 우회율 패널티를 깎아주고 거리를 보정(직선거리 * 1.1x).
  - 공유 도로가 없거나 너무 멀면 패널티 부과(직선거리 * 1.5x).
- **장점:** API 의존성 없음, 개발 공수 낮음, 브라우저 성능 100% 보장.

### 방법 B: 외부 오픈소스 API 렌더링 (OSRM Real-World Routing)
- **개념:** 경로 시퀀스만 내부에서 결정하고, 결정된 5개의 경유지(Waypoints) 좌표를 모아 무료 오픈소스 라우팅 엔진인 **OSRM 데모 서버**에 통신(`fetch`)하여 실제 주행 결과를 받아오는 방식.
- **핵심 로직:**
  - `RouteOptimizer`는 기존처럼 가벼운 방식으로 1차 방문 순서를 최적화.
  - 확정된 최종 순서 좌표들을 모아 `http://router.project-osrm.org/route/v1/driving/{lon},{lat}...` 엔드포인트에 1회 호출.
  - 결과로 반환된 **"진짜 주행 거리(m)"**, 그리고 실제 구불구불한 도로 위를 따라가는 **"경로 폴리라인(GeoJSON)"**을 반환받아 화면에 레이어로 렌더링.
- **장점:** 시각적 임팩트가 압도적(현실 지도처럼 지렁이 이동선 그려짐), 주행거리가 100% 정확(현실 기반).
- **단점:** OSRM 테스트 서버가 느리거나 끊길 수 있음.

## 3. 개발 방향 (결정)
사용자의 요청에 따라, 가장 임팩트가 크면서도 구현이 깔끔한 **[방법 B: OSRM 외부 렌더링 통신]** 을 우선으로 시도합니다.
이후 API 연결 실패 시, **[방법 A]** 로 자연스럽게 우회(Fallback)하도록 설계하는 것이 최적의 아키텍처입니다.

### 작업 내역 (Phase 10)
1. **API 유틸리티 제작**: `src/utils/osrm.ts` 패키지 신설 (위경도 배열 -> API 호출 -> GeoJSON 궤적 반환)
2. **Optimizer 개선**: 순서(TSP) 계산이 끝난 후 비동기(`async`)로 위 유틸을 호출해 최종 거리/요금 재산출.
3. **Layer 개선**: `RouteAnimationLayer.tsx` 가 OSRM이 주는 폴리라인 선형 데이터를 읽어 도로를 따라 자동차가 굽이쳐 그려지도록 애니메이션 보완.

## 4. 버전 관리 정책
기존의 완벽하게 동작하는 코어(Phase 8)를 보호하기 위해, 이 하드코어 작업은 **`feature/real-routing`** 이라는 별도 Git 브랜치에서 진행됩니다. 작업이 실패하거나 마음에 들지 않으면 언제든 `main`으로 안전하게 롤백합니다.
